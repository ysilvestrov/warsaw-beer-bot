# CWS: граційний no-token стан через анонімний catalog-match

**Issue:** GH #245 (серія `chrome-web-store`)
**Дата:** 2026-07-08
**Статус:** design → implementation

## Проблема

Токен розширення видається лише через Telegram-бота (`/extension`). Рев'юер Chrome
Web Store, який ставить розширення «з нуля», токена не має, тож:

- `/match` віддає 401, клієнтський `sendMatch` реджектиться, `runOverlay` **мовчки
  ковтає** помилку (`content/index.ts`) → жодного бейджа, жодного повідомлення;
- popup показує кнопки Refresh/Clear/Sync без натяку, що потрібен токен.

Для рев'юера це виглядає як **non-functional / deceptive** розширення — ризик
відхилення.

## Рішення (у двох словах)

Без токена `/match` працює **анонімно** й повертає лише **глобальні** дані каталогу
(глобальний рейтинг + `untappd_id`), без жодних персональних полів. Бейджі ⭐/⚪
рендеряться як звичайно — розширення одразу корисне «з коробки», ще до будь-якого
налаштування. Popup явно повідомляє, що користувач **не авторизований** і показує
лише глобальні рейтинги, з кнопкою «як отримати токен».

Персональні можливості (✅ «вже пив», власна оцінка, `/enrich/*`, синхронізація
чекінів) лишаються за токеном — вони за визначенням per-user.

Рев'юерський тестовий токен **не потрібен**: анонімний шлях робить основну функцію
повністю перевірюваною без жодного налаштування.

## Архітектура

### 1. Сервер — `/match` стає optional-auth

Зараз `/match` жорстко захищений `authMiddleware` (401 на відсутній/невалідний
токен). Вводимо **окрему optional-auth поведінку саме для `/match`**:

| Заголовок `Authorization` | Поведінка |
|---|---|
| відсутній | анонімно: `telegramId = null`, продовжити |
| `Bearer <valid>` | персонально (як зараз) |
| `Bearer <invalid/unknown>` | **401** (щоб зіпсований токен лишався діагностованим через «Test connection») |

Реалізація: нова middleware (напр. `optionalAuthMiddleware`) або параметр до
наявної. Розбираємо Bearer лише якщо він присутній; присутній-але-невалідний → 401;
відсутній → `telegramId` не встановлюється (null).

`matchRoute`: коли `telegramId` — null, **пропускаємо per-user запити**
(`triedBeerIds`, `latestRatingsByBeer`) і передаємо **порожні** `drunkSet` + `ratings`
у `matchBeerList`. Результат: `matched_beer` з глобальними даними, `is_drunk=false`,
`drunk_uncertain=false`, `user_rating=null`.

`/enrich/*` та `/checkins/*` лишаються **повністю auth-required** — не чіпаємо.

**Тип контексту.** `ApiEnv` наразі, ймовірно, трактує `telegramId` як завжди
присутній; для `/match` він стає `number | null`. Звузити тип/зробити необов'язковим
локально для цього маршруту, не ламаючи enrich/checkins.

### 2. Клієнт — анонімний виклик без токена

- `postMatch` (`extension/src/api/client.ts`): **не додавати** заголовок
  `Authorization`, коли `token` порожній (замість `Bearer ` без значення).
- `handleMatch` (`extension/src/background/index.ts`): **прибрати**
  `if (!token) return { code: 'no-token' }` — завжди викликати `/match`. Порожній
  токен → анонімні результати → бейджі ⭐/⚪ рендеряться штатно.
- Тип `MatchReply` більше не потребує коду `'no-token'` (лишити чи прибрати —
  прибрати, бо жодний шлях його більше не повертає; перевірити споживачів).

Клієнтський код бейджів (`badge.ts`, `content/index.ts`) **не змінюється**: анонімна
відповідь — валідний `MatchResult` з `is_drunk=false`.

### 3. ⏳-бейдж — підтверджено, що не залипає

⏳ (`setSearching`) ставиться **лише** в enrichment (`enrich.ts`), а enrichment
gated на `enrichEnabled && token`. Анонімно enrichment не запускається взагалі, тож
⏳ анонімно не з'являється. У самому enrichment `catch` завжди зводить ⏳→⚪
(`enrich.ts`). Пункт issue «не мають лишатися вічні ⏳» → підтверджуємо тестом/оглядом,
код-фіксу не потребує.

### 4. Popup — стан «не авторизовано»

Коли збережений `token` порожній, popup показує чіткий, неблокуючий рядок:

> ⚠️ Не авторизовано — показуються лише глобальні рейтинги (⭐). Додай токен, щоб
> бачити «вже пив» ✅ і свою оцінку.

+ кнопка **«Як отримати токен»**, що відкриває сторінку Options
(`chrome.runtime.openOptionsPage()`).

Решта кнопок працюють. **Sync** справді потребує користувача: якщо натиснути без
токена — показати, що потрібен токен (перевірити наявну гілку в
`handleCheckinSyncStart`, яка вже пише `outcome:'error'` без токена; дати їй окреме
дружнє повідомлення в `formatSyncStatus`, напр. `not_linked`/новий стан).

### 5. Docs & spec (правила репо — у тому ж PR)

- `spec.md`: описати optional-auth `/match` + анонімну/global-only семантику
  (розділ §4 HTTP API + прапорець у зоні `api_tokens` §3.11).
- `docs/extension-install-uk.md`: додати рамку «без токена бачиш глобальні ⭐;
  з токеном — персональні ✅ і свою оцінку» + оновити опис no-token UX.
- **CWS review notes** — новий `docs/cws-review-notes.md`: що робить розширення,
  різниця анонімний vs авторизований, покроковий сценарій перевірки (без токена).
  Тестовий токен рев'юеру не видаємо.

## Безпека / приватність

- Анонімний `/match` повертає **лише публічні дані каталогу** (глобальний рейтинг,
  `untappd_id`, назва/пивоварня) — жодних персональних полів. Узгоджено з privacy
  policy (#244).
- Ендпоінт стає доступним без токена, але виконує лише catalog-match (той самий, що
  вже був доступний будь-кому з валідним токеном) і вже кооперативно yield-иться
  (`match-list.ts`). Rate-limiting **поза скоупом** (YAGNI): за Cloudflare-тунелем,
  повертає лише публічні дані, CPU кооперативний.

## Тестування (Vitest)

- **Сервер `/match`:**
  - без заголовка → 200 + global-only результати (`is_drunk=false`, `user_rating=null`,
    але `matched_beer.rating_global`/`untappd_id` присутні);
  - валідний токен → персональні результати (регресія);
  - невалідний токен → 401.
- **Клієнт:** `postMatch` не шле `Authorization`, коли токен порожній; шле — коли є.
- **Background:** `handleMatch` повертає результати анонімно (без `no-token`).
- **Popup:** стан «не авторизовано» показується, коли токен порожній; Sync без токена
  дає дружнє повідомлення.

## Поза скоупом

- Рев'юерський тестовий токен (рішення користувача: йдемо повністю анонімним шляхом).
- Rate-limiting анонімного `/match`.
- Діагностика «токен заданий, але невалідний» у popup (лишається в Options →
  «Test connection»).
