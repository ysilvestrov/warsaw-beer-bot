# i18n: підтримка кількох мов інтерфейсу бота

> Статус: **дизайн затверджено 2026-04-27, чекає реалізації**.
> Канонічна спека MVP: [`2026-04-22-warsaw-beer-bot-design.md`](./2026-04-22-warsaw-beer-bot-design.md).

## 1. Мета й not-goals

**Мета.** Бот повинен спілкуватися з користувачем українською, польською або
англійською. Білоруська — окремий етап пізніше. **Російська свідомо не
підтримується**: користувачі з ru-локаллю Telegram отримають англійський
інтерфейс, не український.

**Not-goals (явно поза скоупом):**

- Переклад **сирих даних**: назви пабів, броварень, пива, стилі з ontap.pl
  лишаються тим, що прийшло з джерела.
- Переклад USER-GUIDE.md і design-документації — лишається українською.
- Часові пояси, RTL-мови, складна ICU-розмітка (`select`, `selectordinal`,
  вкладені вирази) — YAGNI на нашому обсязі.
- Hot-swap локалей у середині уже-надісланої сесії редагувань повідомлень —
  на момент `editMessageText` локаль вже зафіксована першим апдейтом.

## 2. Лог рішень

| # | Питання | Вибір |
|---|---------|-------|
| Q1 | Як юзер отримує мову? | Авто-детект з `from.language_code` + override через `/lang`, мова в `user_profiles.language`. |
| Q2 | Які мови в MVP? | UA + PL + EN зараз. BY — окремою задачею пізніше (low-priority issue). |
| Q3 | Що перекладається? | Тільки UI-рядки бота + locale-aware форматування чисел/одиниць. Сирі дані не чіпаємо. |
| Q4 | Інфраструктура? | Hand-rolled TS + `Intl.PluralRules`. Нуль нових залежностей. Compile-time гарантія повноти через TS interface. |
| Q5 | UX `/lang`? | Inline-клавіатура з прапорами (узгоджено з `/filters`). |

```
│   ├── index.ts                       # types, t(), detectLocale, createTranslator
│   ├── locales/
│   │   ├── uk.ts                      # UA — реалізує Messages
│   │   ├── pl.ts                      # PL (Phase 2)
│   │   └── en.ts                      # EN (Phase 2)
│   └── i18n.test.ts
├── bot/
│   ├── middleware/
│   │   └── i18n.ts                    # ← новий: ставить ctx.locale + ctx.t
│   ├── commands/
│   │   └── lang.ts                    # ← нова команда + клавіатура (Phase 2)
│   └── index.ts                       # розширений BotContext, реєстрація middleware
└── storage/
    └── user_profiles.ts               # ← + getUserLanguage / setUserLanguage
```

### 3.2 BotContext

```ts
export type Locale = 'uk' | 'pl' | 'en';   // 'be' додається в Phase 3

export interface BotContext extends Context {
  deps: AppDeps;
  locale: Locale;
  t: Translator;
}

export type Translator = (
  key: keyof Messages,
  params?: Record<string, string | number>,
) => string;
```

Усі pure-формaтери (`route-format.ts`, `newbeers-format.ts`) приймають
`Translator` як параметр — лишаються тестованими в ізоляції.

### 3.3 Middleware

Після `bot.use(deps-injector)` в `bot/index.ts` додаємо:

```ts
const i18n: Middleware<BotContext> = async (ctx, next) => {
  const userId = ctx.from?.id;
  const stored = userId ? getUserLanguage(ctx.deps.db, userId) : null;
  const locale = stored ?? detectLocale(ctx.from?.language_code);
  ctx.locale = locale;
  ctx.t = createTranslator(locale);
  await next();
};
bot.use(i18n);
```

Порядок важливий: middleware читає `ctx.deps.db`, тож має йти після
deps-injector.

## 4. Дані

### 4.1 Міграція v3

```sql
ALTER TABLE user_profiles ADD COLUMN language TEXT;
```

- **Nullable, без DEFAULT.** `NULL` означає «локаль ще не визначена / новий
  юзер». Існуючим користувачам нічого автоматично не присвоюємо — на
  першому апдейті middleware відрезолвить з `language_code` і **запише**
  результат через `setUserLanguage` (це робить наступне визначення
  миттєвим без повторного парсингу).

### 4.2 Storage helpers

```ts
// src/storage/user_profiles.ts
export function getUserLanguage(db: DB, telegramId: number): Locale | null;
export function setUserLanguage(db: DB, telegramId: number, lang: Locale): void;
```

`getUserLanguage` повертає `null`, якщо стовпчик `NULL` **або** містить
рядок, якого нема в `Locale` union (захист від ручних правок БД / даунгрейду
схеми). Middleware у такому разі re-detects.

### 4.3 `detectLocale`

```ts
export function detectLocale(code: string | undefined): Locale {
  const lang = (code ?? '').toLowerCase().split('-')[0];
  if (lang === 'uk') return 'uk';
  if (lang === 'pl') return 'pl';
  // 'be', 'ru', 'en' та все інше — англійська. Особливо явно: ru → en,
  // ми не нав'язуємо UA росіянам, але й не показуємо ru.
  return 'en';
}
```

Коли в Phase 3 додаємо `'be'`, єдина зміна — `if (lang === 'be') return 'be';`
вище за fallback + новий файл локалі.

## 5. API: ключі, plural'и, інтерполяція

### 5.1 Messages interface

```ts
type PluralForms = {
  one?: string;
  few?: string;
  many?: string;
  other: string;       // обов'язковий
};

export interface Messages {
  // app
  'app.start': string;                  // {n_pubs}, {n_beers}
  'app.no_interesting': string;

  // link / import
  'link.success': string;               // {username}
  'link.invalid': string;
  'import.progress': string;            // {done}, {total}
  'import.done': string;                // {imported}, {skipped}

  // newbeers
  'newbeers.empty': string;
  'newbeers.more_pubs': string;         // {extra}

  // route
  'route.preparing': string;            // {count}
  'route.no_data': string;
  'route.matrix_progress': string;      // {cached}, {total}, {missing}
  'route.fill_missing': string;         // {done}, {total}
  'route.searching_tour': string;
  'route.failed': string;
  'route.header': string;               // {count}, {km}, {pubs}

  // pluralized counters
  'pubs.in_route': PluralForms;         // count

  // refresh
  'refresh.starting': string;
  'refresh.ontap_progress': string;     // {done}, {total}, {slug}
  'refresh.untappd_progress': string;   // {done}, {total}, {username}
  'refresh.done': string;

  // filters
  'filters.title': string;
  'filters.reset': string;

  // lang
  'lang.prompt': string;
  'lang.changed': string;               // {locale_native_name}
}
```

Точний перелік сформується в PR 1 (зараз ~30 ключів, точна цифра проявиться
при міграції call-сайтів). Контракт лишається: один інтерфейс — три файли
локалей.

### 5.2 Pluralization

```ts
// Усередині createTranslator(locale):
const pr = new Intl.PluralRules(locale);

function t(key, params) {
  const raw = locales[locale][key];
  if (typeof raw === 'string') return interpolate(raw, params);
  // PluralForms — pivot завжди params.count (стандарт ICU/i18next)
  const form = pr.select(params?.count as number);   // 'one'|'few'|'many'|'other'
  const template = raw[form] ?? raw.other;
  return interpolate(template, params);
}
```

### 5.3 Інтерполяція

`{name}` — простий regex-replace на `params.name`. ICU-розмітка не потрібна.

```ts
function interpolate(tmpl: string, params?: Record<string, string|number>): string {
  if (!params) return tmpl;
  return tmpl.replace(/\{(\w+)\}/g, (_, k) => String(params[k] ?? `{${k}}`));
}
```

Якщо параметр відсутній → залишаємо плейсхолдер у виводі (легше зловити в
тесті, ніж тихий `undefined`).

### 5.4 Locale-aware форматери

Виносимо в `src/i18n/index.ts`:

```ts
export function fmtAbv(locale: Locale, abv: number | null): string;
// uk/pl → '6,1%', en → '6.1%'

export function fmtKm(locale: Locale, meters: number): string;
// uk/pl → '14,4 км', en → '14.4 km'
```

`newbeers-format.ts` і `route-format.ts` починають використовувати ці
варіанти замість локальних реалізацій.

## 6. Touchpoints

Інвентар user-facing рядків — ~14 reply-сайтів у командах + 4 формaтери.
Усі зачіпаються в PR 1, але точково.

### 6.1 Команди

- `start.ts` — привітання + інструкція `/link`. Найбільший блок (~10 рядків).
- `link.ts` — успіх / помилки парсингу.
- `import.ts` — прогрес-меседжі + підсумок.
- `newbeers.ts` — fallback `app.no_interesting`.
- `route.ts` — прогрес-рядки, помилки, `route.preparing` хедер.
- `filters.ts` — заголовок + label `Скинути`. Стилі (IPA/Pils/Stout/Sour)
  лишаються Latin per Q3 — це жаргон, не переклад.
- `refresh.ts` — прогрес-меседжі.

### 6.2 Формaтери

- `route-format.ts` — `route.header`, `pubs.in_route` (plural).
- `newbeers-format.ts` — `newbeers.more_pubs` суфікс. `⭐ —` лишається
  універсальним.

### 6.3 Команда `/lang` (Phase 2)

- Без аргументу → `ctx.replyWithHTML(t('lang.prompt'), keyboard)` з кнопками
  🇺🇦 Українська | 🇵🇱 Polski | 🇬🇧 English.
- callback `lang:uk|pl|en` → `setUserLanguage(db, telegram_id, lang)` →
  `editMessageText(t('lang.changed', { locale_native_name: ... }))`. Текст
  підтвердження вже на новій локалі (бо `ctx.t` пере-створюється для нового
  меседжа в наступному апдейті — тут редагуємо одразу, тому викликаємо
  `createTranslator(newLocale)('lang.changed')`).

## 7. Тестування

### 7.1 Юніт-тести

1. **`detectLocale`** — таблично всі мапінги: `uk`/`uk-UA`→`uk`, `pl`/`pl-PL`→`pl`,
   `ru`/`ru-RU`→`en`, `be`/`be-BY`→`en` (Phase 1), `en`/`en-US`→`en`,
   `undefined`→`en`, `''`→`en`, `'qq'`→`en`.
2. **`t()` core** — інтерполяція (включно з відсутнім параметром →
   плейсхолдер у виводі), plural-селекція, fallback на `other` коли форма
   відсутня.
3. **Повнота локалей** — runtime-тест:
   ```ts
   expect(Object.keys(uk).sort()).toEqual(Object.keys(en).sort());
   expect(Object.keys(uk).sort()).toEqual(Object.keys(pl).sort());
   ```
   Catches: пропущені ключі (TS теж ловить, але runtime — додатковий guard
   на порожні рядки `''`).
4. **Інтерполяція params** — на 5-7 ключових повідомлень: snapshot-тест
   `t('route.header', { count: 10, km: 14.4, pubs: 6 })` для всіх трьох
   локалей.
5. **Locale-aware `fmtAbv` / `fmtKm`** — UA/PL → `6,1%` / `14,4 км`,
   EN → `6.1%` / `14.4 km`.

### 7.2 Інтеграційний тест на middleware

Mock `ctx.from.language_code` + DB → `i18nMiddleware` ставить `ctx.locale`
правильно. Чотири кейси:

1. Збережена локаль у БД → береться вона (ignore `language_code`).
2. Немає в БД, є `language_code: 'pl-PL'` → `pl`, **і запис у БД**.
3. Немає в БД, `language_code: 'ru'` → `en`, **і запис у БД**.
4. Невпізнаваний `language_code` → `en`.

### 7.3 E2E (post-merge, ручна)

Чек-ліст у PR-описі: пройтися всіма командами на UA, PL, EN. Жодного
hardcoded UA/EN рядка, що пройшов повз `t()`.

## 8. Поетапна доставка

### PR 1 — `feat/i18n-foundation`

Pure refactor. Жодних змін поведінки бота.

- `src/i18n/` модуль (types, `t`, `detectLocale`, `Intl.PluralRules`).
- Міграція v3 (`user_profiles.language` nullable).
- `i18nMiddleware`.
- `BotContext` розширений `locale` + `t`.
- **Усі** hardcoded UA-рядки винесено в `src/i18n/locales/uk.ts`. `Locale =
  'uk'` (тільки!). PL/EN ще нема — ані файлів, ані command'и `/lang`.
- Тести: `detectLocale`, `t()`, plural UA, інтерполяція, locale-aware
  `fmtAbv`/`fmtKm`, middleware.

**Критерій ОК:** бот стартує, всі команди дають той самий вивід байт-у-байт,
що до PR. `language_code` у БД зберігається як `'uk'` для всіх (бо інша
локаль ще недоступна).

### PR 2 — `feat/i18n-pl-en-lang`

Фактичне розширення на три мови.

- `Locale = 'uk' | 'pl' | 'en'`.
- `src/i18n/locales/pl.ts` + `en.ts` — повні переклади (workflow див. §9).
- Команда `/lang` + `keyboards.ts` extension.
- `setUserLanguage`.
- USER-GUIDE: додаємо секцію про `/lang` (сама USER-GUIDE лишається UA per
  not-goals).
- Spec §1-§3 канонічної дизайн-доки оновлюється коротким згадуванням i18n
  (memory rule про sync доки).

**Критерій ОК:** перемикання `/lang` міняє мову інтерфейсу одразу для
наступного апдейту. Всі три локалі покривають однаковий набір ключів.

### Phase 3 (окремою задачею, low-priority issue)

Додаємо `'be'`:

- `Locale = 'uk' | 'pl' | 'en' | 'be'`.
- `src/i18n/locales/be.ts`.
- `detectLocale` — `'be' → 'be'`.
- Кнопка 🇧🇾 у `/lang` клавіатурі.

## 9. Workflow перекладів

### 9.1 Артефакт

Один markdown-файл на локаль у `docs/i18n/`:

- `docs/i18n/pl-translations-review.md`
- `docs/i18n/en-translations-review.md`
- (Phase 3) `docs/i18n/be-translations-review.md`

### 9.2 Структура файлу

```markdown
# PL translations review

Status: draft  (draft → reviewed-by-user → reviewed-by-llm → final)

## Команди

| Key | Source (uk) | Draft (pl) | Notes |
|-----|-------------|------------|-------|
| `app.start` | Привіт! Я допоможу… | Cześć! Pomogę… | |
| `link.success` | Профіль прив'язано: {username} | Profil powiązany: {username} | |

## Plural'и

| Key | Form | uk | pl |
|-----|------|----|----|
| `pubs.in_route` | one | паб у маршруті: {count} | pub w trasie: {count} |
| `pubs.in_route` | few | паби у маршруті: {count} | puby w trasie: {count} |
| `pubs.in_route` | many | пабів у маршруті: {count} | pubów w trasie: {count} |

## Декоратори

ABV separator: `,` (uk/pl), `.` (en)
Distance unit: `км` / `km`
```

### 9.3 Цикл (per-locale, у PR 2)

1. **Я генерую draft.** Відкриваю PR з `docs/i18n/<locale>-translations-review.md`
   + `src/i18n/locales/<locale>.ts`. Markdown і `.ts` синхронні (одне
   джерело істини для review — markdown; `.ts` — runtime). `Status: draft`.
2. **Користувач вичитує** в IDE/PR. Виправляє очевидне inline. → `Status:
   reviewed-by-user`.
3. **Користувач передає markdown іншій LLM** з промптом «перевір PL-переклади
   на тон, ідіоматичність, граматику; дай оновлений markdown або diff».
4. **Користувач повертає виправлення** (коментар у PR / новий комміт у
   гілку / paste у чат). Я оновлюю **markdown і `.ts` одночасно**. →
   `Status: reviewed-by-llm`.
5. **Фінальна вичитка користувача** після LLM-правок. → `Status: final`.
   Мерджимо.

### 9.4 Sync expectation

Під час review markdown — джерело істини; обидва файли (`.md` + `.ts`)
оновлюються разом у тій же гілці перед мерджем PR. Після `Status: final`
markdown лишається як snapshot review-циклу; майбутні точкові правки
рядків ідуть напряму в `.ts`, без оновлення markdown'а — він знадобиться
наступного разу, коли захочемо зробити повний review (нова мова, масштабна
зміна tone).

CI-guard, що парсить markdown і звіряє з `.ts`, **навмисно не вводимо** —
overhead не виправдовує користі на 60 рядках. Альтернатива «генерувати
`.ts` з `.md`» теж відхилена з тієї ж причини.

## 10. Ризики

| Ризик | Вплив | План Б |
|-------|-------|--------|
| Telegram змінює формат `from.language_code` | низький | `detectLocale` приймає `string \| undefined`, фолбек на `en` для всього невпізнаного. |
| LLM-перевірка ламає тон UA-оригіналу | середній | Послідовний цикл §9.3 + фінальна вичитка користувача. |
| Користувач ставить локаль через `/lang`, але після рестарту юзера зникає | низький | Тест #7.2.1 покриває «збережена локаль у БД має пріоритет над `language_code`». |
| Десинк markdown і `.ts` після правок | низький | §9.4 — runtime-guard у тестах. |
| Plural-форми у UA для `count: 0` | низький | `Intl.PluralRules('uk').select(0) === 'many'` — стандартна категорія, тести §7.1.2 фіксують. |

## 11. Майбутні розширення (поза скоупом)

- Локалізація USER-GUIDE.md (поки що — UA only).
- Додаткові локалі (DE, CZ — місцеві сусіди).
- Більш складна ICU-розмітка, якщо з'явиться кейс на `select` (ґендеризовані
  фрази, відмінкові форми зі складною логікою).
- Локалізація beer-style buttons у `/filters` (наразі IPA/Pils/Stout/Sour
  лишаються Latin універсально).
